# Nash Consolidate — Promote Learnings into Skills

You are a knowledge manager for the Nash design skills. Your job is to review accumulated learnings and evolve the skills based on what's been learned.

## When to Run

- Weekly (recommended) — keeps skills fresh
- When `/nash-reflect` flags 3+ entries on the same topic
- Before a major project kicks off — ensure skills have latest knowledge
- When the user feels skills are making outdated suggestions

## Execution Steps

### Step 1: Inventory all learnings

Read all `learnings.md` files across skills:

```
skills/nash-ingest/learnings.md
skills/nash-locate/learnings.md
skills/nash-clarify/learnings.md
skills/nash-propose/learnings.md
skills/nash-spec/learnings.md
skills/nash-tasks/learnings.md
skills/nash-prototype/learnings.md
skills/nash-govern/learnings.md
improve/learnings/nash-patterns.md
improve/learnings/design-system.md
improve/learnings/component-selections.md
```

Also read `improve/learnings/INDEX.md` for the full chronological log.

### Step 2: Analyze learnings per skill

For each skill's learnings, categorize:

**Promotable** (→ merge into SKILL.md):
- 3+ HIGH-confidence entries on the same topic
- Rules that apply to every invocation of the skill
- Corrections that were validated multiple times
- Component selection patterns that are now standard

**Keep** (→ stays in learnings.md):
- MEDIUM-confidence entries with < 3 occurrences
- Context-specific patterns (useful but not universal)
- Recent entries that haven't been validated yet

**Archive** (→ move to learnings-archive.md):
- LOW-confidence observations that are > 30 days old without reinforcement
- Entries superseded by newer, more specific learnings
- One-off findings that weren't repeated

**Remove**:
- Duplicates (keep the most specific version)
- Contradicted entries (the newer correction wins)
- Stale entries about code that no longer exists

### Step 3: Promote learnings into skills

For each promotable learning:

1. **Read the current SKILL.md** for the target skill
2. **Find the right section** to insert the learning:
   - Component selection rules → add to the component reference section
   - Do/Don't rules → add to the execution rules or quality rules section
   - Pattern knowledge → add to the patterns or examples section
3. **Draft the edit** — convert the learning into skill instruction format:

**Before (in learnings.md):**
```markdown
### 2026-04-21 CORRECTION HIGH — Sheet not Dialog for editing
**Learning**: Always use Sheet for edit flows in Nash portal, not Dialog
**Context**: Proposed Dialog for order editing, user corrected
**Example**:
- DO: `<Sheet>` side panel for edit forms
- DON'T: `<Dialog>` modal for edit forms
```

**After (promoted to nash-propose/SKILL.md):**
```markdown
### Nash-Specific Rules
- **Edit flows**: Always use `<Sheet>` (side panel), never `<Dialog>` (modal) for editing data in Nash portal. Dialog is only for confirmations and destructive action warnings.
```

4. **Show the proposed edit** to the user for approval before applying
5. **Apply the edit** to SKILL.md
6. **Mark the learning as promoted** in learnings.md:
```markdown
### 2026-04-21 CORRECTION HIGH — Sheet not Dialog for editing ✅ PROMOTED 2026-04-28
```

### Step 4: Build the consolidated knowledge base

Create or update `improve/learnings/consolidated.md` — the "Nash Design Bible" that captures everything learned:

```markdown
# Nash Design Knowledge Base

> Auto-generated by /nash-consolidate. Last updated: [date]
> This is the consolidated view of all validated learnings.

---

## Component Selection Rules

### When to use Sheet vs Dialog
- **Sheet**: All edit flows, detail views, forms. Side panel pattern.
- **Dialog**: Confirmations, destructive action warnings, quick choices only.
- *Source: 5 corrections across 3 sessions*

### When to use DataTable vs Table
- **DataTable** (@usenash/components): When you need pagination, sorting, filtering, column visibility
- **Table** (@usenash/ui): Simple, small, static data display (< 10 rows)
- *Source: 3 validated patterns*

---

## Do's and Don'ts

### DO
- Use semantic tokens: `bg-primary`, `text-muted-foreground`
- Use flex/gap for spacing between siblings
- Use `createDialogStore()` for sheet/dialog state
- Reference existing features as precedents in proposals

### DON'T
- Hardcode colors: `bg-[#e01a22]`, `text-gray-400`
- Use margins between siblings (use gap instead)
- Use Dialog for edit flows (use Sheet)
- Use arbitrary Tailwind values: `p-[13px]`

---

## Validated Patterns

### [Pattern Name]
**When**: [Situation]
**Approach**: [What to do]
**Example feature**: [Nash feature that demonstrates this]
**Validated**: [N] times across [N] sessions

---

## Component Selection Examples

| Need | Best choice | Source | Why | Learned from |
|------|------------|--------|-----|-------------|
| Edit form panel | Sheet | @usenash/ui | Nash standard | 5 corrections |
| Data with pagination | DataTable | @usenash/components | Built-in features | 3 patterns |

## Architecture Preferences

| Area | Nash Preference | Avoid | Reason |

| Client state | Zustand | Redux/Context | createDialogStore pattern |
```
